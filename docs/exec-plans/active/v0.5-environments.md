# Execution Plan: v0.5 ‚Äî Environments

**Goal**: `hearth build my-api --repo github.com/user/my-api` clones a repo, runs setup commands, and snapshots the result. `hearth shell my-api` restores it in ~50ms with services running. `hearth claude my-api` gives you Claude Code inside that environment.

**Spec**: `docs/product-specs/environments.md`

---

## Prerequisites

- v0.4 dm-thin snapshots (for fast restore)
- Existing `hearth claude` command (refactored to use environments internally)

---

## Phase 1: Hearthfile Parsing

### 1.1 TOML parser [no deps]

Add `smol-toml` as a dependency (zero-dep, ESM, <10KB).

`src/environment/hearthfile.ts`:

```typescript
interface Hearthfile {
  name: string;
  repo?: string;
  branch?: string;
  workdir?: string;            // default: /home/agent/<repo-name>
  setup?: string[];
  start?: string[];
  ports?: number[];
  ready?: string;              // URL to poll
  files?: Array<{
    from: string;              // host path (~ expanded)
    to: string;                // guest path
    mode?: string;             // e.g. "0600"
  }>;
  github_token_env?: string;   // host env var name
}
```

- `parseHearthfile(path: string): Hearthfile` ‚Äî read and validate TOML
- `findHearthfile(dir: string): string | null` ‚Äî look for `Hearthfile.toml` in dir
- Validate: `name` is required, must match snapshot name rules (`/^[a-zA-Z0-9_-]+$/`)
- Validate: `ports` are valid port numbers
- Validate: `files[].from` paths exist on host (at build time)

### 1.2 Environment metadata [no deps]

`src/environment/metadata.ts`:

```typescript
interface EnvironmentMeta {
  name: string;
  repo?: string;
  branch?: string;
  builtAt: string;             // ISO 8601
  hearthfile: Hearthfile;      // full config, needed for start phase
}
```

- `writeEnvironmentMeta(snapshotDir: string, meta: EnvironmentMeta)` ‚Äî writes `environment.toml`
- `readEnvironmentMeta(snapshotDir: string): EnvironmentMeta | null` ‚Äî returns null if not an environment
- `isEnvironment(snapshotName: string): boolean` ‚Äî check for `environment.toml` in snapshot dir

## Phase 2: GitHub Authentication

### 2.1 Token resolution [no deps]

`src/environment/github.ts`:

- `resolveGitHubToken(hearthfile: Hearthfile): string | null`
- Resolution order:
  1. `hearthfile.github_token_env` ‚Üí read that env var from host
  2. `GITHUB_TOKEN` env var on host
  3. `gh auth token` subprocess (if `gh` is on PATH)
  4. `null` (public repos only)
- Token is injected into the VM during build as a git credential helper:
  ```bash
  git config --global credential.helper '!f() { echo "password=$TOKEN"; }; f'
  ```
- Token is **not** baked into the snapshot ‚Äî credential helper is cleared before snapshotting

### 2.2 SSH key injection [no deps]

Handled by the `files` block in the Hearthfile ‚Äî no special code needed. The generic file injection in Phase 3 covers this. Just document the pattern:

```toml
[[files]]
from = "~/.ssh/id_ed25519"
to = "/home/agent/.ssh/id_ed25519"
mode = "0600"
```

## Phase 3: Build Command

### 3.1 Build orchestration [depends: 1.1, 1.2, 2.1]

`src/environment/build.ts`:

`buildEnvironment(hearthfile: Hearthfile): Promise<void>`

Sequence:
1. Boot fresh sandbox via daemon
2. `sandbox.enableInternet()`
3. Inject GitHub token (if resolved) via `sandbox.exec(git config ...)`
4. Inject `files` from Hearthfile ‚Äî expand `~`, `sandbox.writeFile()` or `sandbox.upload()`, set permissions via `sandbox.exec(chmod ...)`
5. Clone repo: `sandbox.exec("git clone [--branch <branch>] <repo> <workdir>")`
6. Run each `setup` command sequentially via `sandbox.exec()`, with `cwd` set to workdir
   - Stream stdout/stderr to host terminal so user sees progress
   - If any command fails (non-zero exit), abort build with error
7. Clear git credential helper: `sandbox.exec("git config --global --unset credential.helper")`
8. Snapshot as `hearthfile.name`
9. Write `environment.toml` into the snapshot directory
10. Destroy sandbox

### 3.2 CLI: `hearth build` [depends: 3.1]

`src/cli/build.ts`:

```
hearth build [name]
  --file <path>       Path to Hearthfile.toml (default: ./Hearthfile.toml)
  --repo <url>        Repo URL (overrides Hearthfile, enables no-Hearthfile flow)
  --branch <branch>   Branch (overrides Hearthfile)
```

Behavior:
- If `--repo` provided without a local Hearthfile: clone the repo first into a temp sandbox, check for `Hearthfile.toml` in the repo root, use it if found
- If `--repo` provided and no Hearthfile found anywhere: build a minimal environment (clone only, no setup)
- `name` argument overrides `hearthfile.name`
- If snapshot with that name already exists, error with "already exists, use `hearth rebuild`"
- Print timing: `Built my-api in 34.2s (snapshot: 2.1GB)`

### 3.3 CLI: `hearth rebuild` [depends: 3.1]

`src/cli/rebuild.ts`:

- Read `environment.toml` from existing snapshot to get the Hearthfile
- Delete existing snapshot
- Re-run build with the stored Hearthfile
- Accepts same `--branch` override for switching branches

## Phase 4: Start Phase

### 4.1 Environment-aware restore [depends: 1.2]

`src/environment/start.ts`:

`startEnvironment(name: string): Promise<Sandbox>`

Sequence:
1. Restore sandbox from snapshot (existing `fromSnapshot` path)
2. Read `environment.toml` to get the Hearthfile
3. Re-inject credentials (GitHub token, files from `files` block) ‚Äî credentials are never snapshotted
4. If `start` commands defined, run each sequentially with `cwd` set to workdir
   - Stream output to terminal
   - If any command fails: print error, but **don't abort** ‚Äî continue to shell so user can debug
5. If `ready` URL defined, poll it (HTTP GET, 200 = ready, retry every 500ms, timeout 30s)
   - If timeout: print warning, continue anyway
6. If `ports` defined, call `sandbox.forwardPort()` for each, print the host mappings
7. Return the sandbox

### 4.2 Update `hearth shell` [depends: 4.1]

`src/cli/shell.ts` ‚Äî modify existing command:

- If `name` is provided and is an environment (has `environment.toml`): use `startEnvironment(name)` instead of bare `fromSnapshot(name)`
- If `name` is provided and is a bare snapshot: current behavior (just restore)
- If no `name`: current behavior (fresh sandbox)
- Set working directory for the shell to the environment's `workdir`

### 4.3 Update `hearth claude` [depends: 4.1]

`src/cli/claude.ts` ‚Äî modify existing command:

```
hearth claude [name] [-- claude-args...]
```

- If `name` is an environment: `startEnvironment(name)`, then layer Claude credentials + config on top (existing injection logic)
- If `name` omitted: current behavior (claude-base snapshot)
- Claude Code's working directory set to the environment's `workdir`

## Risks

- **Build time UX**: Long builds (large repos, slow installs) need visible progress. Streaming setup command output to the terminal is essential ‚Äî a silent 2-minute wait would feel broken.
- **GitHub token security**: Token is injected into the VM during build. Must be cleared before snapshotting. If the build crashes mid-way, the snapshot could contain the token. Mitigation: clear credentials as the first step in the snapshot sequence, not the last step of build.
- **Large snapshots**: A repo with a big `node_modules` or `target/` directory inflates the rootfs. dm-thin CoW helps (only changed blocks stored per sandbox), but the base snapshot can still be multi-GB. Not a blocker for v0.5 ‚Äî squashfs layers address this later.
- **Start command races**: If `start` commands launch background processes, subsequent commands might run before the process is ready. The `ready` URL check helps for HTTP services, but non-HTTP services (databases, message queues) need their own readiness pattern. For v1, document that users should handle this in their start scripts (e.g., `redis-server --daemonize yes && redis-cli ping`).
