### 1.2 macOS M3+ via Lima [DONE ‚Äî 2026-03-17]

Fully automated via `hearth lima setup/start/stop/status/teardown`. Tested end-to-end on M4 Pro.

See `docs/exec-plans/active/v0.3-lima-macos.md` for the full execution plan and `docs/references/lima-integration-learnings.md` for implementation learnings.

**What shipped:**
- `hearth lima setup` ‚Äî creates Lima VM with nested virt, provisions Node.js + Docker, runs `hearth setup` inside VM, starts daemon
- `hearth lima start/stop` ‚Äî lifecycle management with KVM permission fixup and stale socket cleanup
- `hearth lima status/teardown/shell` ‚Äî inspection and cleanup commands
- Platform detection (`src/platform.ts`) ‚Äî chip detection, macOS version, Lima status
- Prebuilt agent binaries (aarch64 + x86_64) ‚Äî hosted on GitHub releases, eliminates Zig requirement
- `setitimer` fix for aarch64 agent (replaces x86_64-only `alarm` syscall)
- Lima `portForwards` for Unix socket sharing (guest ‚Üí host via SSH tunnel)
- `HEARTH_DIR` + `HEARTH_DAEMON_SOCK` env vars for cross-VM path configuration
- `DaemonClient` + `RemoteSandbox` exported from package index
- macOS guard on `hearth setup` directing users to `hearth lima setup`

**Key learnings:**
- Unix sockets don't work across virtiofs ‚Äî use Lima's native `portForwards` config
- Lima homedir ‚âÝ macOS homedir ‚Äî need env var overrides
- KVM/Docker group membership doesn't apply until re-login ‚Äî use sudo for setup, udev rule for daemon
- Rootfs tar extraction fails on virtiofs ‚Äî use local tmpdir

### 1.3 macOS M1/M2 via remote daemon [documented]

M1/M2 users SSH tunnel to a Linux host running `hearth daemon`. No code changes needed ‚Äî the daemon and SSH tunneling work today.

## Phase 2: Observability

### Goal

Agents can observe what happened inside a sandbox ‚Äî logs, metrics, resource usage.

See `docs/design-docs/observability.md` for the full design.

### 2.1 Agent metric collection [no deps]

Extend the Zig agent to scrape `/proc` on a 5s interval:
- `/proc/stat` (CPU), `/proc/meminfo` (memory), `/proc/diskstats` (disk)
- `/proc/[pid]/stat` for per-process stats
- Send to host over a new vsock port as JSON

### 2.2 Agent log collection [no deps]

Extend the Zig agent to tail `/dev/kmsg` and `/dev/log`:
- Ship structured JSON log entries to host over vsock

### 2.3 Victoria Logs + Metrics on host [depends: 2.1, 2.2]

- Download during `npx hearth setup`
- Shared instance, data labeled by `sandbox_id`
- Delete data on `sandbox.destroy()`

### 2.4 SDK query API [depends: 2.3]

```typescript
const obs = await sandbox.observe();
const logs = await sandbox.logs.query({ filter: "level:error", since: "5m" });
const ok = await sandbox.metrics.check("process_resident_memory_bytes < 400000000");
```
