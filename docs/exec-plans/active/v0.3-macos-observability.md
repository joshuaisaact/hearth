# Execution Plan: v0.3 — Platform Support, Observability, npm Publish

## What's already done (from v0.2)

- Daemon server + client — tested and working on Linux (28 tests passing)
- Request ID correlation, serial dispatch, sandbox cleanup on disconnect
- Shared `encodeMessage`/`parseFrames` framing utilities

## Phase 1: Platform Support

### 1.1 WSL2 support [no code needed]

WSL2 has native `/dev/kvm` support. Hearth works as-is inside WSL2.

### 1.2 macOS via remote daemon [documented]

macOS users SSH tunnel to a Linux host running `hearth daemon`. The daemon and SSH tunneling work today. Lima was evaluated but removed — the virtiofs + nested KVM overhead added too much latency for interactive use.

### 1.3 Prebuilt agent binaries [DONE]

- Cross-compiled hearth-agent for x86_64-linux and aarch64-linux
- Hosted as GitHub release assets (`agent-v0.2.0`)
- `hearth setup` downloads prebuilt first, falls back to Zig build from source

## Phase 2: Observability

### Goal

Agents can observe what happened inside a sandbox — logs, metrics, resource usage.

See `docs/design-docs/observability.md` for the full design.

### 2.1 Agent metric collection [no deps]

Extend the Zig agent to scrape `/proc` on a 5s interval:
- `/proc/stat` (CPU), `/proc/meminfo` (memory), `/proc/diskstats` (disk)
- `/proc/[pid]/stat` for per-process stats
- Send to host over a new vsock port as JSON

### 2.2 Agent log collection [no deps]

Extend the Zig agent to tail `/dev/kmsg` and `/dev/log`:
- Ship structured JSON log entries to host over vsock

### 2.3 Victoria Logs + Metrics on host [depends: 2.1, 2.2]

- Download during `npx hearth setup`
- Shared instance, data labeled by `sandbox_id`
- Delete data on `sandbox.destroy()`

### 2.4 SDK query API [depends: 2.3]

```typescript
const obs = await sandbox.observe();
const logs = await sandbox.logs.query({ filter: "level:error", since: "5m" });
const ok = await sandbox.metrics.check("process_resident_memory_bytes < 400000000");
```
