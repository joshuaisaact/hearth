# Execution Plan: v0.4 — dm-thin Instant Snapshots

**Goal**: Snapshot restore drops from ~1.3s (file copy on virtiofs/ext4) to <50ms on all platforms by using device-mapper thin provisioning for block-level CoW.

**Spec**: `docs/design-docs/squashfs-overlays.md` (dm-thin is an alternative/complement to the squashfs approach)

**Why dm-thin over squashfs**: squashfs solves distribution (shippable environments), dm-thin solves performance (instant snapshots). dm-thin is the immediate win — it makes Lima/ext4 as fast as btrfs reflinks. Squashfs can be layered on top later for the distribution story.

---

## Background

Current snapshot restore copies three files per sandbox:
- `rootfs.ext4` (~2GB)
- `vmstate.snap` (~10KB)
- `memory.snap` (~512MB)

On btrfs/XFS with reflinks, the rootfs copy is O(1). On ext4 and virtiofs (Lima on macOS), it's a real 2GB+ copy — the dominant cost in the 1.3s restore time.

dm-thin provisioning does block-level CoW: create a thin snapshot of the base volume, and only blocks that are written get allocated. Creating a snapshot is a metadata operation (~1ms).

## Phase 1: Thin Pool Setup

### 1.1 Pool creation during `hearth setup` [no deps]

After the base rootfs is built, create a thin pool:

```bash
# Create a sparse file for the pool (e.g. 20GB, only allocates on write)
truncate -s 20G ~/.hearth/thin-pool.img

# Set up the loopback device
losetup --find --show ~/.hearth/thin-pool.img  # → /dev/loop0

# Create the thin pool (data + metadata in same device for simplicity)
dmsetup create hearth-pool \
  --table "0 $(blockdev --getsz /dev/loop0) thin-pool /dev/loop0 /dev/loop0 128 0"
```

- Sparse file means it only uses disk for blocks actually written
- Pool size configurable (default 20GB, enough for ~40 sandboxes with typical overlay usage)
- Requires root — `hearth setup` already runs with sudo inside Lima

### 1.2 Import base rootfs into pool [depends: 1.1]

```bash
# Create thin volume 0 (the base)
dmsetup message hearth-pool 0 "create_thin 0"

# Activate it
dmsetup create hearth-base \
  --table "0 $ROOTFS_SECTORS thin /dev/mapper/hearth-pool 0"

# Copy rootfs into the thin volume
dd if=~/.hearth/bases/ubuntu-24.04.ext4 of=/dev/mapper/hearth-base bs=1M
```

One-time operation during setup. The base volume holds the golden rootfs image.

### 1.3 Detection and fallback [no deps]

`src/vm/thin.ts`:

- `isThinPoolAvailable()` → boolean (check for `/dev/mapper/hearth-pool`)
- `ensureThinPool()` → create pool if not exists
- Fall back to current file-copy approach if dm-thin is not available (no root, WSL2 without device-mapper, etc.)

## Phase 2: Thin Snapshots for Sandboxes

### 2.1 Snapshot-based sandbox creation [depends: 1.2]

Replace the `copyFile(rootfs)` in `restoreFromDir()` with:

```bash
# Allocate next thin ID
dmsetup message hearth-pool 0 "create_snap $NEW_ID $BASE_ID"

# Activate as a device
dmsetup create hearth-sb-$SANDBOX_ID \
  --table "0 $SECTORS thin /dev/mapper/hearth-pool $NEW_ID"
```

This is ~1ms instead of ~1s. The sandbox gets its own CoW device that shares all blocks with the base until written.

Firecracker config changes:
- `putDrive("rootfs", "/dev/mapper/hearth-sb-$SANDBOX_ID", ...)` instead of a file path

### 2.2 Sandbox destroy [depends: 2.1]

```bash
dmsetup remove hearth-sb-$SANDBOX_ID
dmsetup message hearth-pool 0 "delete $THIN_ID"
```

Instant deallocation — all CoW blocks returned to the pool.

### 2.3 User snapshots [depends: 2.1]

`sandbox.snapshot(name)` creates a thin snapshot of the sandbox's thin volume:

```bash
dmsetup message hearth-pool 0 "create_snap $SNAP_ID $SANDBOX_THIN_ID"
```

`Sandbox.fromSnapshot(name)` creates a thin snapshot of the snapshot. Snapshots of snapshots — it's CoW all the way down.

vmstate.snap and memory.snap are still files (small, ~512MB total). Only the rootfs benefits from dm-thin.

## Phase 3: Integration

### 3.1 Update Sandbox class [depends: 2.1-2.3]

- `restoreFromDir()` checks `isThinPoolAvailable()`
- If available: create thin snapshot, pass device path to Firecracker
- If not: fall back to current `copyFile` with `COPYFILE_FICLONE`
- `destroySync()` cleans up thin volume
- `snapshot()` creates thin snapshot + copies vmstate/memory files

### 3.2 Update daemon [depends: 3.1]

No changes needed — daemon calls Sandbox methods, thin provisioning is transparent.

### 3.3 Update `hearth setup` and `hearth lima setup` [depends: 1.1]

- `hearth setup` creates thin pool if running as root
- `hearth lima setup` creates thin pool inside Lima (always has root)
- Skip thin pool if no root — fall back to file-copy mode with a message

### 3.4 Pool management CLI [depends: 1.1]

- `hearth pool status` — show pool usage, thin volume count
- `hearth pool compact` — reclaim unused space (fstrim on the pool device)
- `hearth pool destroy` — tear down pool and loopback device

## Expected Performance

| Operation | Current (Lima/ext4) | Current (btrfs) | With dm-thin |
|-----------|-------------------|-----------------|--------------|
| Sandbox create | ~1.3s | ~135ms | ~50ms |
| Snapshot create | ~1s | ~100ms | ~5ms |
| Snapshot restore | ~1.3s | ~130ms | ~50ms |
| Destroy | ~50ms | ~50ms | ~5ms |

The ~50ms for create/restore with dm-thin is dominated by Firecracker snapshot load + agent reconnect, not the rootfs copy.

## Risks

- **WSL2 compatibility**: Microsoft's WSL kernel may not have `CONFIG_DM_THIN_PROVISIONING`. Need to test. Fallback to file-copy is the safety net.
- **Root requirement**: dm-thin needs root. The daemon inside Lima already has sudo. On Linux, users would need to run the daemon with capabilities or sudo. This is a change from the current "no root needed for sandbox" model.
- **Loop device limits**: Linux defaults to 8 loop devices. Can be increased but may surprise users. Could use a real block device instead of loopback for production.
- **Pool exhaustion**: if the pool fills up, sandbox writes fail. Need monitoring and a clear error message. `hearth pool status` helps here.
