# Execution Plan: v0.3 — macOS Support + Observability

## Phase 1: macOS Support via Lima

### Goal

`npx hearth setup` works on macOS. The user doesn't need to know about the Linux VM underneath.

### Architecture

```
macOS host
┌──────────────────────────────┐
│ Hearth SDK (TypeScript)      │
│   Sandbox.create()           │
│   sandbox.exec()             │
│         │                    │
│         │ SSH / Unix socket  │
│         ▼                    │
│ Lima VM (Linux, KVM)         │
│ ┌──────────────────────────┐ │
│ │ Hearth daemon            │ │
│ │   Firecracker            │ │
│ │   Guest VMs (sandboxes)  │ │
│ │   vsock channels         │ │
│ └──────────────────────────┘ │
└──────────────────────────────┘
```

The SDK detects macOS and transparently proxies all operations into a Lima VM where Firecracker runs natively with KVM.

### Implementation Steps

#### 1.1 Lima VM provisioning [no deps]

`npx hearth setup` on macOS:
- Check for Lima (`limactl --version`), prompt to install via `brew install lima` if missing
- Create a Hearth-specific Lima instance: `limactl create --name hearth` with a minimal Ubuntu template
- Configure: 2+ CPUs, 4GB+ RAM, mount `~/.hearth` as a shared directory
- Enable nested virtualization (required for Firecracker inside Lima)
- Start the instance if not running

Lima config (`hearth.yaml`):
```yaml
vmType: vz  # Apple Virtualization.framework (fast on M-series)
cpus: 4
memory: 4GiB
disk: 20GiB
mounts:
  - location: "~/.hearth"
    writable: true
provision:
  - mode: system
    script: |
      # Install Firecracker + deps inside the Lima VM
      # This runs the same setup as Linux but inside the VM
```

#### 1.2 Transport layer [depends: 1.1]

The SDK needs to communicate with the Hearth daemon inside the Lima VM. Options:

**Option A: SSH exec**
- `limactl shell hearth -- node dist/cli/hearth.js setup` for setup
- SDK calls all operations by SSH-exec'ing commands into the Lima VM
- Simple but adds latency per operation

**Option B: Shared filesystem + Unix socket**
- Lima mounts `~/.hearth` into the VM
- Hearth daemon inside the VM listens on a Unix socket at `~/.hearth/daemon.sock`
- SDK on macOS connects to the socket directly (Lima exposes host-mounted sockets)
- Lower latency, cleaner

**Option C: TCP socket**
- Daemon listens on a TCP port inside the VM
- Lima forwards the port to localhost
- Simplest networking but requires port allocation

**Recommendation**: Option B. Lima's shared filesystem means the Unix socket is accessible from macOS. This gives us the daemon backend we deferred from v0.2, with Lima as the transport.

#### 1.3 Daemon backend [depends: 1.2]

A simple daemon that runs inside the Lima VM and exposes the Sandbox API over a Unix socket:

- Accepts JSON-RPC or simple length-prefixed JSON commands
- Maps to the existing Sandbox class methods (create, exec, upload, etc.)
- Manages VM lifecycle, cleanup, snapshots
- The macOS SDK becomes a thin client that serializes method calls and sends them to the daemon

This daemon also benefits Linux users who want multi-process access to sandboxes (the v0.2 exec plan mentioned this as a future item).

#### 1.4 SDK platform detection [depends: 1.3]

```typescript
// On Linux: direct mode (current behavior)
const sandbox = await Sandbox.create();

// On macOS: transparently proxies through Lima VM
// Same API, same code — platform detected internally
const sandbox = await Sandbox.create();
```

`Sandbox.create()` checks `process.platform`:
- `linux`: use DirectBackend (current code)
- `darwin`: use LimaBackend → connects to daemon inside Lima VM

### What Lima handles for us

- KVM access (Virtualization.framework provides nested virt on Apple Silicon)
- Filesystem sharing (mount `~/.hearth` for snapshots, rootfs, etc.)
- Network forwarding (Lima auto-forwards ports)
- VM lifecycle (start/stop/restart)

### What we need to build

- Lima instance provisioning in `npx hearth setup`
- A thin daemon that wraps the Sandbox API
- A client backend that talks to the daemon
- Platform detection in the SDK

### Open questions

1. **Lima vs OrbStack**: OrbStack is faster but commercial. Lima is open source. Default to Lima, document OrbStack as an alternative?
2. **Shared filesystem performance**: Lima's virtiofs mount has good performance but adds latency for large file operations. Snapshots and rootfs copies happen inside the VM, so this mostly affects upload/download from the Mac host.
3. **Lima VM resource allocation**: How much CPU/RAM to give the Lima VM? It needs to run Firecracker VMs inside it. 4 CPU / 4GB RAM as default?
4. **First-run experience**: The Lima VM takes ~30-60 seconds to create on first run. How do we communicate this to the user?

## Phase 2: Observability

### Goal

Agents can observe what happened inside a sandbox — logs, metrics, resource usage — not just stdout/stderr.

### Implementation

See `docs/design-docs/observability.md` for the full design. Summary:

#### 2.1 Agent metric collection [no deps]

Extend the Zig agent to collect from `/proc`:
- CPU usage (`/proc/stat`)
- Memory usage (`/proc/meminfo`)
- Disk I/O (`/proc/diskstats`)
- Per-process stats (`/proc/[pid]/stat`)
- Push to host over vsock on a configurable interval (default 5s)

#### 2.2 Agent log collection [no deps]

Extend the Zig agent to tail:
- `/dev/kmsg` (kernel messages)
- `/dev/log` (syslog socket)
- Ship to host over vsock as structured JSON

#### 2.3 Victoria Logs + Metrics on host [depends: 2.1, 2.2]

- Download Victoria Logs + Victoria Metrics during `npx hearth setup`
- Start lazily on first `Sandbox.create()`
- Data labeled with `sandbox_id`
- Delete data on `sandbox.destroy()`

#### 2.4 SDK query API [depends: 2.3]

```typescript
// Quick snapshot of what's happening
const obs = await sandbox.observe();
// { cpu: 23.5%, memory: 142MB/512MB, recentLogs: [...], errors: [...] }

// Query logs
const logs = await sandbox.logs.query({ filter: "level:error", since: "5m" });

// Query metrics (PromQL)
const cpu = await sandbox.metrics.query({ query: "rate(cpu_usage[1m])" });

// Boolean check for agent assertions
const ok = await sandbox.metrics.check('process_resident_memory_bytes < 400000000');
```

## Phase 3: npm Publish

- Ship prebuilt hearth-agent binaries for x86_64 and aarch64 (avoid requiring Zig on user's machine)
- Package as `hearth` on npm
- Postinstall script runs `npx hearth setup` or prompts user
- README on npm registry

---

## Priority Order

1. **macOS via Lima** — unblocks ~50% of developers
2. **npm publish** — unblocks everyone else (currently requires cloning the repo)
3. **Observability** — the long-term differentiator, but not blocking any use case

## Deferred to v0.4+

- TAP networking for raw TCP/UDP (beyond HTTP/HTTPS proxy)
- GPU passthrough (Firecracker limitation)
- Python SDK
- Template marketplace (share snapshots)
- Allowlist for internet proxy (security)
