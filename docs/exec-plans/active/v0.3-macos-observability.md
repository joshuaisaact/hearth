# Execution Plan: v0.3 — Platform Support, Observability, npm Publish

## What's already done (from v0.2)

- Daemon server + client — tested and working on Linux (28 tests passing)
- Request ID correlation, serial dispatch, sandbox cleanup on disconnect
- Shared `encodeMessage`/`parseFrames` framing utilities

## Phase 1: Platform Documentation + WSL2

### 1.1 WSL2 support [no code needed]

WSL2 has native `/dev/kvm` support. Hearth works as-is inside WSL2. Only documentation needed (done — see `docs/design-docs/platform-support.md` and README).

### 1.2 macOS M3+ via Lima [documented, needs testing]

Lima with `nestedVirtualization: true` gives KVM inside the VM. The daemon runs inside Lima, macOS SDK connects via shared filesystem socket.

**Needs Mac with M3+ to test.** The code path is:
1. User creates Lima instance with nested virt
2. Runs `npx hearth setup` inside Lima
3. Runs `hearth daemon` inside Lima
4. macOS SDK connects to `~/.hearth/daemon.sock` via DaemonClient

All the code for this exists (daemon server/client). What's missing:
- A `hearth setup --lima` command that automates Lima instance creation
- Integration testing on actual M3+ hardware

### 1.3 macOS M1/M2 via remote daemon [documented]

M1/M2 users SSH tunnel to a Linux host running `hearth daemon`. No code changes needed — the daemon and SSH tunneling work today.

## Phase 2: Observability

### Goal

Agents can observe what happened inside a sandbox — logs, metrics, resource usage.

See `docs/design-docs/observability.md` for the full design.

### 2.1 Agent metric collection [no deps]

Extend the Zig agent to scrape `/proc` on a 5s interval:
- `/proc/stat` (CPU), `/proc/meminfo` (memory), `/proc/diskstats` (disk)
- `/proc/[pid]/stat` for per-process stats
- Send to host over a new vsock port as JSON

### 2.2 Agent log collection [no deps]

Extend the Zig agent to tail `/dev/kmsg` and `/dev/log`:
- Ship structured JSON log entries to host over vsock

### 2.3 Victoria Logs + Metrics on host [depends: 2.1, 2.2]

- Download during `npx hearth setup`
- Shared instance, data labeled by `sandbox_id`
- Delete data on `sandbox.destroy()`

### 2.4 SDK query API [depends: 2.3]

```typescript
const obs = await sandbox.observe();
const logs = await sandbox.logs.query({ filter: "level:error", since: "5m" });
const ok = await sandbox.metrics.check("process_resident_memory_bytes < 400000000");
```

## Phase 3: npm Publish

### 3.1 Prebuilt agent binaries [no deps]

- CI builds hearth-agent for x86_64-linux and aarch64-linux
- Binaries stored as npm package assets or downloaded during `npx hearth setup`
- Removes Zig as a user requirement

### 3.2 Package configuration [depends: 3.1]

- `npm publish` as `hearth` package
- `bin.hearth` points to CLI entry
- Postinstall: prompt user to run `npx hearth setup`
