# Execution Plan: v0.2 ŌĆö Usable by Others

**Goal**: Someone else can `npm install hearth`, run `npx hearth setup`, and use the SDK without manual rootfs/kernel/binary setup.

**Secondary goal**: Networking works ŌĆö agents can start a web server in a sandbox and hit it from the host.

---

## Phase 1: `npx hearth setup` CLI

### 1.1 CLI entrypoint [no deps]
`src/cli/setup.ts`

- `npx hearth setup` command (add `bin` to package.json)
- Detect host architecture (`x86_64` / `aarch64`)
- Check `/dev/kvm` access, print actionable error
- Report filesystem type at `~/.hearth/` (btrfs/XFS = reflink, ext4 = full copy)
- Idempotent ŌĆö skip already-downloaded components

### 1.2 Firecracker download [depends: 1.1]
- Download Firecracker v1.15.0 tarball from GitHub releases
- Verify SHA256
- Extract `firecracker` + `jailer` to `~/.hearth/bin/`

### 1.3 Kernel download [depends: 1.1]
- Download guest kernel from Firecracker CI S3
- Store at `~/.hearth/bases/vmlinux`

### 1.4 Rootfs build [depends: 1.1]
- Download Ubuntu squashfs from Firecracker CI S3
- Unsquash, patch in hearth-agent, create init script
- Pack as ext4 at `~/.hearth/bases/ubuntu-24.04.ext4`
- Requires: `unsquashfs` and `mkfs.ext4` on PATH (document in error message)
- Alternative: use Docker if available (fallback path)

### 1.5 Agent binary [depends: 1.1]
- Ship prebuilt hearth-agent for x86_64 and aarch64
- Store at `~/.hearth/bin/hearth-agent-{arch}`
- Bake into rootfs during 1.4

### 1.6 Base snapshot creation [depends: 1.2, 1.3, 1.4]
- After all assets are ready, create the base snapshot
- Runs the same `ensureBaseSnapshot()` logic
- Reports: "Base snapshot created. Sandbox.create() will now restore in ~135ms"

## Phase 2: Networking

### 2.1 TAP device creation [no deps]
`src/network/tap.ts`

- Create TAP device for each sandbox
- Requires: `ip` command or CAP_NET_ADMIN
- Assign IP to TAP device (host side of the veth pair)
- Configure guest IP via kernel boot args or DHCP

### 2.2 NAT / masquerade [depends: 2.1]
`src/network/nat.ts`

- Set up iptables MASQUERADE for sandbox outbound traffic
- Sandboxes can reach the internet via host's network
- Cleanup on sandbox destroy

### 2.3 Port forwarding [depends: 2.1]
`src/network/forward.ts`

- `sandbox.forwardPort(guestPort)` ŌåÆ returns `{ host: "127.0.0.1", port: <allocated> }`
- iptables DNAT from host port to guest IP:guestPort
- Cleanup on sandbox destroy

### 2.4 Network config in Firecracker [depends: 2.1]
- Add `putNetworkInterface()` call to VM boot sequence
- Update snapshot creation to include network interface
- Guest gets network on snapshot restore

### 2.5 SDK API [depends: 2.3]
```typescript
const { host, port } = await sandbox.forwardPort(8080);
// curl http://localhost:{port} hits guest:8080
```

## Phase 3: Spawn (streaming exec)

### 3.1 Agent protocol extension [no deps]
- Add `spawn` method to agent protocol
- Returns stdout/stderr chunks as they arrive (not buffered)
- Host receives length-prefixed stream messages: `{type: "stdout", data: "..."}`

### 3.2 SDK API [depends: 3.1]
```typescript
const proc = sandbox.spawn("npm run dev");
proc.stdout.on("data", (chunk) => console.log(chunk));
await proc.wait(); // or proc.kill()
```

## Phase 4: Upload/Download ŌĆö DONE

Implemented via tar streaming over vsock CONNECT (port 1026).
- `sandbox.upload(hostPath, guestPath)` ŌĆö host tar c | vsock | guest tar x
- `sandbox.download(guestPath, hostPath)` ŌĆö guest tar c | vsock | host tar x
- Uses busybox tar in guest, system tar on host
- Agent uses shared `startListener`, `readLine`, `toPathZ`, `execWithFdRedirect` helpers
