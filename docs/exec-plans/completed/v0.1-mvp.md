# Execution Plan: v0.1 MVP — COMPLETED 2026-03-16

**Goal**: A working `Sandbox.create()` → `exec()` → `destroy()` loop on a local Linux machine with KVM.

**Success criteria**: Run this and get "hello world" back:
```typescript
import { Sandbox } from "hearth";

const sandbox = await Sandbox.create();
const result = await sandbox.exec("echo hello world");
console.log(result.stdout); // "hello world\n"
await sandbox.destroy();
```

---

## Phase 1: Setup CLI (`npx hearth setup`)

Downloads and caches all binaries needed to run sandboxes.

### 1.1 Binary downloader [no deps]
`src/setup/download.ts`

- Download file from URL with progress reporting
- SHA256 verification after download
- Extract .tgz archives
- Cache to `~/.hearth/bin/`

### 1.2 Firecracker setup [depends: 1.1]
`src/setup/firecracker.ts`

- Download Firecracker v1.15.0 release tarball for host arch (x86_64/aarch64)
- Verify SHA256 checksum
- Extract `firecracker` and `jailer` binaries to `~/.hearth/bin/`
- Verify `/dev/kvm` access, print actionable error if missing

### 1.3 Kernel + rootfs setup [depends: 1.1]
`src/setup/images.ts`

- Download guest kernel from Firecracker CI S3 bucket
- Download Ubuntu squashfs rootfs from Firecracker CI S3 bucket
- Unsquash, patch in hearth-agent binary, repack as ext4
- Store at `~/.hearth/bases/vmlinux-*` and `~/.hearth/bases/ubuntu-*.ext4`

### 1.4 Setup CLI entrypoint [depends: 1.2, 1.3]
`src/setup/index.ts`

- `npx hearth setup` runs all setup steps
- Reports what was downloaded, filesystem type (reflink support), KVM status
- Idempotent — skips already-downloaded files

## Phase 2: Guest Agent (Zig)

Port from flint's agent.zig. The agent runs inside the VM and handles commands from the host.

### 2.1 Core agent: vsock + exec [no deps]
`agent/src/main.zig`

- Connect to host over AF_VSOCK (CID 2, port 1024) with retry-on-connect
- Main loop: read length-prefixed JSON request, dispatch, write response
- `exec` method: fork + execve("/bin/sh", "-c", cmd), capture stdout/stderr via pipes, timeout via SIGALRM
- `health` method: respond with uptime
- Static buffers, zero allocation (port from flint)

### 2.2 File I/O methods [depends: 2.1]
`agent/src/main.zig`

- `write_file`: write base64-decoded content to path, create parent dirs, set mode
- `read_file`: read file, base64-encode, return in response
- Path length validation, file size limits

### 2.3 Build system [depends: 2.1]
`agent/build.zig`

- Cross-compile for x86_64-linux and aarch64-linux
- ReleaseSafe, stripped
- Output to `agent/bin/hearth-agent-{arch}`

## Phase 3: VM Lifecycle (TypeScript)

Spawn Firecracker, configure it, boot a VM.

### 3.1 Binary resolver [no deps]
`src/vm/binary.ts`

- Find firecracker binary: check `~/.hearth/bin/`, then PATH
- Find kernel + rootfs: check `~/.hearth/bases/`
- Throw actionable errors if missing ("run npx hearth setup")

### 3.2 Firecracker API client [no deps]
`src/vm/api.ts`

- HTTP client over Unix socket (Node's `http.request` with `socketPath`)
- Methods matching Firecracker REST API:
  - `putMachineConfig(vcpus, memMb)`
  - `putBootSource(kernel, bootArgs)`
  - `putDrive(id, path, isRoot, isReadOnly)`
  - `putNetworkInterface(id, hostDev, guestMac)`
  - `putVsock(cid, udsPath)`
  - `putAction(actionType)`
  - `patchVm(state)` (Pause/Resume)
  - `putSnapshot(snapshotPath, memPath)`
- Typed responses, error handling

### 3.3 VM spawn + configure [depends: 3.1, 3.2]
`src/vm/vm.ts`

- Spawn `firecracker` process with `--api-sock` pointed at a temp Unix socket
- Wait for API socket to appear
- Configure: machine config → boot source → rootfs drive → vsock → start
- Track child process PID for cleanup
- Kill on destroy (SIGTERM, then SIGKILL after timeout)

### 3.4 Rootfs overlay [depends: 3.1]
`src/vm/rootfs.ts`

- On sandbox create: `cp --reflink=auto` base rootfs to a temp per-VM path
- On sandbox destroy: delete the temp rootfs
- Temp files in `~/.hearth/run/{sandbox_id}/`

### 3.5 vsock CID allocation [no deps]
`src/vm/cid.ts`

- Allocate unique CID (3+) for each VM
- Lockfile-based: read/write `~/.hearth/lock` to prevent collisions between concurrent processes
- Release CID on sandbox destroy

## Phase 4: Agent Client (TypeScript)

Host-side client that talks to the guest agent over vsock.

### 4.1 vsock connection [no deps]
`src/agent/vsock.ts`

- Connect to Firecracker's vsock UDS (`{api_sock_dir}/v.sock_{port}`)
- Retry with backoff (agent may not be ready immediately after VM boot)
- Send/receive length-prefixed JSON messages

### 4.2 Agent client [depends: 4.1]
`src/agent/client.ts`

- `exec(command, opts?)` → `{ stdout, stderr, exitCode }`
- `writeFile(path, content, mode?)` → `void`
- `readFile(path)` → `string | Buffer`
- `health()` → `{ uptimeMs }`
- Timeout handling per-request
- Typed errors (AgentError, TimeoutError)

---

## Phase 5: Sandbox API

The public-facing API. Glues everything together.

### 5.1 Sandbox class [depends: 3.3, 3.4, 3.5, 4.2]
`src/sandbox/sandbox.ts`

- `Sandbox.create(config?)` — static factory:
  1. Allocate CID
  2. Copy rootfs overlay
  3. Spawn + configure Firecracker VM
  4. Connect agent client (wait for health check)
  5. Return Sandbox instance
- `exec(command, opts?)` — delegate to agent client
- `writeFile(path, content)` — delegate to agent client
- `readFile(path)` — delegate to agent client
- `destroy()` — kill VM, delete overlay, release CID
- `Symbol.asyncDispose` — for `await using` support

### 5.2 Types + errors [no deps]
`src/sandbox/types.ts`

- `SandboxConfig`: `{ template?, vcpus?, memoryMb?, diskSizeMb? }`
- `ExecResult`: `{ stdout, stderr, exitCode }`
- `ExecOptions`: `{ cwd?, env?, timeout? }`

`src/errors.ts`

- `HearthError` (base)
- `VmBootError`, `ExecError`, `TimeoutError`, `AgentError`, `ResourceError`

### 5.3 Public exports [depends: 5.1, 5.2]
`src/index.ts`

- Export `Sandbox`, types, errors
- This is already stubbed

## Phase 6: Smoke Test

### 6.1 Integration test [depends: 5.1]
`src/sandbox/sandbox.test.ts`

- Requires KVM — skip in CI without `/dev/kvm`
- Test: create sandbox, exec "echo hello", assert stdout, destroy

## What actually shipped (beyond plan)

- **Snapshots**: Added during v0.1 (was planned for v0.2). Base snapshot created on first use, restore in ~135ms.
- **Code review**: Extracted shared `waitForFile`/`errorMessage` to util.ts, fixed deadlock risk in Zig agent pipe reads (poll()), async parallel file copies, capped stderrBuf, parallel FC API config calls.
- **Removed `SandboxConfig`**: vcpus/memoryMb were silently ignored. Removed until snapshot-per-config is implemented.

## Deferred to v0.2

- Networking (TAP, port forwarding, NAT)
- Observability (Victoria Logs/Metrics, sandbox.observe())
- VM pool / daemon backend
- `npx hearth setup` CLI (currently manual setup)
- `npx hearth setup --template alpine`
- OTel trace collection
- Spawn (streaming exec)
- Upload/download (directory transfer)
- Configurable VM resources (vcpus, memory) — needs per-config snapshots

                                                   └─ 3.5 CID alloc

Phase 4 (agent client)   Phase 5 (sandbox)      Phase 6 (test)
 4.1 vsock conn            5.2 types+errors       6.1 integration test [5.1]
  └─ 4.2 agent client      5.1 Sandbox [3.3, 3.4, 3.5, 4.2]
                             └─ 5.3 exports [5.1, 5.2]

Phases 1, 2, 3 can be developed in parallel. Phase 4 needs the vsock protocol from Phase 2 but not the binary. Phase 5 integrates everything. Phase 6 is end-to-end validation.
