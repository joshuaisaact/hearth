# Execution Plan: v0.3 Lima — macOS M3+ Support

**Goal**: A macOS M4 user runs `hearth lima setup`, waits a few minutes, and has a working local sandbox environment via the daemon.

**Spec**: `docs/product-specs/lima-macos-support.md`

**Prerequisites**: Daemon server + client already working (v0.2). Platform support design doc written.

---

## Phase 0: Prebuilt Agent Binary

### 0.1 Cross-compile agent for aarch64-linux [no deps]

The Zig agent already supports cross-compilation via `standardTargetOptions`. Build:

```bash
cd agent && zig build -Dtarget=aarch64-linux
```

Produces `zig-out/bin/hearth-agent` (~800KB static binary).

### 0.2 Host binary as GitHub release asset [depends: 0.1]

Create a GitHub release (e.g. `agent-v0.1.0`) with two assets:
- `hearth-agent-x86_64-linux`
- `hearth-agent-aarch64-linux`

### 0.3 Update `hearth setup` to download prebuilt agent [depends: 0.2]

Update `setupAgent()` in `src/cli/setup.ts`:
- Detect architecture (`x86_64` / `aarch64`)
- Download from GitHub release instead of building with Zig
- Fallback: if `zig` is on PATH, build from source (for development)
- This eliminates Zig as a user requirement on both Linux and Lima

## Phase 1: Platform Detection + Guard Rails

### 1.1 Platform detection utility [no deps]

`src/platform.ts`

- `getPlatform()` → `"linux" | "macos" | "wsl"` (uses `os.platform()`, checks for WSL via `/proc/version`)
- `getChipSupportsNestedVirt()` → boolean (parse `sysctl machdep.cpu.brand_string` or `system_profiler SPHardwareDataType` for M3/M4)
- `getMacosVersion()` → major version number (parse `sw_vers -productVersion`)
- `isLimaInstalled()` → boolean (check `limactl` on PATH)
- `getLimaInstanceStatus(name)` → `"running" | "stopped" | "not-found"` (parse `limactl list --json`)

### 1.2 Guard `hearth setup` on macOS [depends: 1.1]

Update `src/cli/setup.ts`:

- Before `checkKvm()`, check platform
- If macOS: print message directing to `hearth lima setup` and exit
- If WSL or Linux: proceed as today

## Phase 2: Lima Instance Management

### 2.1 Lima config generation [depends: 1.1]

`src/cli/lima.ts`

- `generateLimaConfig(opts: { cpus?: number; memory?: string })` → YAML string
- Ubuntu 24.04 cloud image (aarch64)
- Defaults: 4 CPU, `"4GiB"` RAM, 50 GiB disk — overridable via `--cpus` and `--memory`
- `nestedVirtualization: true`
- Writable mount for `~/.hearth`
- Provision script: Node.js 22, Docker, KVM permissions (no Zig — agent is prebuilt)

### 2.2 `hearth lima setup` [depends: 2.1]

Orchestrates the full first-time setup:

1. **Preflight checks**: macOS? M3+? macOS 15+? Lima installed?
2. **Create instance**: Write lima.yaml to temp file, `limactl create --name hearth <tempfile>`
3. **Start instance**: `limactl start hearth`
4. **Wait for provisioning**: Lima runs the provision script automatically on first start
5. **Run hearth setup inside VM**: `limactl shell hearth -- npx hearth setup` (or direct node invocation)
6. **Start daemon**: `limactl shell hearth -- hearth daemon` (backgrounded)
7. **Verify**: Check daemon socket exists at `~/.hearth/daemon.sock`, send ping via DaemonClient
8. **Print success** with SDK usage example

Error handling at each step — if any step fails, print what happened and how to recover.

### 2.3 `hearth lima start` [depends: 2.1]

1. Check Lima instance exists (if not: "Run `hearth lima setup` first")
2. `limactl start hearth` (no-op if already running)
3. Start daemon inside VM: `limactl shell hearth -- hearth daemon` (backgrounded via nohup)
4. Verify daemon socket accessible from macOS
5. Print status

### 2.4 `hearth lima stop` [no deps]

1. Kill daemon inside VM: `limactl shell hearth -- pkill -f "hearth daemon"` (graceful)
2. `limactl stop hearth`
3. Print status

### 2.5 `hearth lima status` [depends: 1.1]

1. Check Lima instance state via `limactl list --json`
2. Check daemon socket exists and is responsive (connect + ping)
3. Print: VM state, daemon state, resource usage, active sandbox count

### 2.6 `hearth lima teardown` [no deps]

1. Confirm with user (destructive — deletes VM and all data inside it)
2. Stop daemon if running
3. `limactl delete hearth --force`
4. Print status

Note: `~/.hearth/` on macOS host survives teardown (it's a shared mount, not inside the VM). Snapshots and config persist.

## Phase 3: CLI Wiring

### 3.1 Add `lima` subcommand to CLI [depends: 2.2–2.6]

Update `src/cli/hearth.ts`:

```
hearth lima setup      Create and provision Lima VM
hearth lima start      Start Lima VM and daemon
hearth lima stop       Stop daemon and Lima VM
hearth lima status     Show Lima VM and daemon status
hearth lima teardown   Destroy Lima VM
hearth lima shell      Open a shell inside the Lima VM
```

Parse `process.argv[2] === "lima"` then delegate to `process.argv[3]` subcommand.

### 3.2 DaemonClient default socket path [no deps]

Update `src/daemon/client.ts`:

- `client.connect()` with no args defaults to `~/.hearth/daemon.sock`
- Saves macOS users from needing to know the path

### 3.3 Export DaemonClient from package index [no deps]

Update `src/index.ts`:

```typescript
export { DaemonClient, RemoteSandbox } from "./daemon/client.js";
```

macOS users import `DaemonClient` directly — it should be a first-class export.

## Phase 4: Testing

### 4.1 Platform detection unit tests [depends: 1.1]

`src/platform.test.ts` — mock `os.platform()`, test chip detection parsing, Lima status parsing.

### 4.2 Lima integration test [depends: all above, requires M3+ Mac]

Manual test on actual M4 hardware:

1. Fresh Mac — no Lima, no Hearth
2. `brew install lima`
3. `npx hearth lima setup` — should complete end-to-end
4. `DaemonClient.connect()` → `create()` → `exec("echo hello")` → works
5. `hearth lima stop` → `hearth lima start` → daemon reconnects
6. `hearth lima teardown` → clean state

### 4.3 Error path testing [depends: 4.2]

- Run on M1/M2 simulator (or mock): get clear error about chip support
- Run without Lima installed: get install instructions
- Kill daemon mid-operation: client gets clean error
- Kill Lima VM mid-operation: sandbox cleanup works
